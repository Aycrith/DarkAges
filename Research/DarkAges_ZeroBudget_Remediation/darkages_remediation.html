ADR-001: VPS Selection

Decision: Hetzner AX42 for production, OVH VPS-3 for development.

Rationale: AX42 offers the best price/performance ratio at €49/month with 64GB RAM and high single-thread performance (critical for game servers). OVH VPS-3 at $15/month is sufficient for development and early testing.

Trade-offs: Hetzner requires EU data center acceptance; OVH has less predictable CPU performance due to virtualization.

ADR-002: Database Consolidation

Decision: Replace Redis+ScyllaDB with PostgreSQL for Phase 0.

Rationale: For under 200 concurrent players, PostgreSQL single instance can handle all hot-state and persistence needs. Simplifies operations, eliminates network overhead between data stores.

Migration Path: When exceeding 500 concurrent players, reintroduce Redis for hot-state caching. ScyllaDB only needed at 1000+ players with high write throughput requirements.

3.1 Serialization Strategy

FlatBuffers vs Protocol Buffers vs Godot StreamPeerBuffer

Serialization Performance Comparison
LibrarySerializationDeserializationSizeZero-Copy

StreamPeerBufferManualManualCustomNo
ADR-003: Serialization Strategy

Decision: Keep FlatBuffers for server-side, use Godot's ByteBuffer for client.

Rationale: FlatBuffers' zero-copy deserialization is critical for server performance at 60Hz tick rate. Godot client can use ByteBuffer with manual serialization to avoid GDExtension complexity.

Implementation: Generate FlatBuffers schema for both C++ and GDScript (via flatc --gen-json or custom codegen).

GameNetworkingSockets GDExtension Binding

For Godot 4 integration with GameNetworkingSockets, the recommended approach is:

  1. Option A (Recommended): Use Godot's built-in ENetMultiplayerPeer for initial implementation. ENet provides reliable UDP with channel support sufficient for MMO needs.
  2. Option B (Advanced): Create a thin GDExtension wrapper around GameNetworkingSockets. Reference implementation pattern available in godot-cpp examples.

3.2 Deterministic Simulation

Fixed-Point Arithmetic Libraries

Fixed-Point Library Comparison
LibraryTypePrecisionGodot Compatible

FixedMath.NetC#ConfigurableGodot Mono only
nilpunch/fixed-pointC#64-bit configurableUnity/Godot

ADR-004: Deterministic Math

Decision: Use custom int64_t fixed-point (32.32) implementation in C++ server.

Rationale: Cross-platform determinism requires controlling the entire math pipeline. Custom implementation avoids external dependencies and ensures consistent behavior across x86/ARM.

Implementation: Store positions as int64_t (32.32 fixed), convert to float only for Godot rendering. This maintains server authority while allowing visual smoothness.

Cross-Platform Determinism Checklist

  • Use fixed-point arithmetic for all physics/position calculations
  • Avoid floating-point in serialized state
  • Use deterministic random number generator (xoroshiro128+)
  • Compile with -ffloat-store on GCC to prevent extended precision
  • Test on both x86_64 and ARM64 regularly

ADR-005: ECS Framework

Decision: Continue using EnTT, but adopt group/stable iteration patterns.

Rationale: EnTT is already integrated. The benchmark shows significant performance gains using groups (4-5x faster than runtime views). Template complexity can be managed with wrapper functions.

Refactoring: Replace runtime views with stable groups for position/velocity queries:

// Before (runtime view - slower)
auto view = registry.view<Position, Velocity>();

// After (stable group - faster)
auto group = registry.group<Position, Velocity>(entt::get<>);
group.each([](auto& pos, auto& vel) { ... });

5.1 Movement Validation

Server-Side Rewind (Lag Compensation)

// Pseudocode for server-authoritative movement with rewind
class MovementValidator {
    static constexpr float MAX_SPEED = 6.0f;      // m/s
    static constexpr float MAX_SPRINT = 9.0f;     // m/s
    static constexpr float TELEPORT_THRESHOLD = 10.0f;  // m
    static constexpr int REWIND_BUFFER_MS = 200;  // Max RTT/2

    struct PositionSnapshot {
        uint32_t timestamp;
        Vector3 position;
        Vector3 velocity;
    };

    std::deque<PositionSnapshot> history_buffer;

public:
    bool validate_movement(uint32_t client_time, 
                          Vector3 claimed_pos,
                          Vector3 claimed_vel) {
        // Calculate server time at client action
        uint32_t server_time = get_server_time();
        int32_t rtt = estimate_rtt(player_id);
        uint32_t action_time = server_time - (rtt / 2);

        // Rewind to historical state
        Vector3 historical_pos = interpolate_position(action_time);

        // Validate movement from historical position
        float max_distance = MAX_SPEED * (rtt / 1000.0f + TICK_RATE);
        float actual_distance = (claimed_pos - historical_pos).length();

        if (actual_distance > max_distance * 1.2f) {  // 20% tolerance
            log_suspicious(player_id, "speed violation");
            return false;
        }

        return true;
    }
};

5.2 Statistical Detection

Heuristic Anti-Cheat Implementation

class HeuristicAntiCheat {
    struct PlayerStats {
        float avg_reaction_time;
        float input_variance;
        uint32_t impossible_actions;
        uint32_t packet_timing_violations;
    };

    std::unordered_map<PlayerID, PlayerStats> stats;

public:
    void record_input(PlayerID id, uint32_t timestamp, InputState input) {
        auto& s = stats[id];

        // Detect inhuman reaction times (< 100ms consistently)
        float reaction = calculate_reaction_time(id, timestamp);
        if (reaction < 0.1f) {
            s.impossible_actions++;
        }

        // Detect perfect input patterns (bots)
        float variance = calculate_input_variance(id, input);
        if (variance < 0.01f && s.input_variance > 100) {
            flag_for_review(id, "suspicious consistency");
        }

        // Packet timing analysis
        if (check_packet_timing_violation(id, timestamp)) {
            s.packet_timing_violations++;
        }

        // Threshold-based actions
        if (s.impossible_actions > 10) {
            kick_player(id, "suspicious gameplay patterns");
        }
    }
};

6.2 Hot Reload

C++ Hot Reload via Shared Libraries

// Host executable
#include <dlfcn.h>

typedef void (*TickFunc)(float delta, void* state);

int main() {
    void* state = nullptr;
    void* handle = nullptr;
    TickFunc tick = nullptr;
    time_t last_mtime = 0;

    while (true) {
        // Check if library changed
        struct stat st;
        if (stat("libgame.so", &st) == 0 && st.st_mtime != last_mtime) {
            if (handle) dlclose(handle);

            handle = dlopen("./libgame.so", RTLD_NOW);
            tick = (TickFunc)dlsym(handle, "game_tick");
            last_mtime = st.st_mtime;

            printf("Hot reload completed\n");
        }

        if (tick) {
            state = tick(1.0f/60.0f, state);
        }

        std::this_thread::sleep_for(16ms);
    }
}
